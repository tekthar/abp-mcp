# Design: abp-mcp — Auto-Generate MCP Servers from ABP Applications

Generated by /office-hours on 2026-04-24
Repo: abp-mcp (new project)
Status: APPROVED
Mode: Builder (Open Source Passion Project)

## Problem Statement

ABP Framework apps already declare their business logic as `IApplicationService` contracts with typed DTOs, permission attributes, XML documentation, and multi-tenancy awareness. That's richer metadata than any OpenAPI/Swagger description. Today, exposing an ABP backend to AI agents (Claude, Copilot, Cursor) means either:

1. Hand-writing MCP tools per service — tedious, error-prone, doesn't scale.
2. Pointing a generic OpenAPI→MCP generator at the Swagger endpoint — produces low-quality MCP servers where agents get confused by too many tools with mechanical descriptions.

`abp-mcp` is a NuGet package that adds `builder.Services.AddAbpMcp()` to any ABP host, inspects the app's Application Services at startup, and exposes them as a permission-aware MCP endpoint inside the same process. No sidecar. No separate deployment. No OpenAPI middleman.

## What Makes This Cool

**The framing is the moat.** Every OpenAPI→MCP tool ships mechanical, low-quality servers. The industry is learning this the hard way (Neon's "Yay or Nay" post). `abp-mcp` skips the OpenAPI layer and generates from ABP's Application Service abstraction — one level higher, with richer semantics. That's a genuinely differentiated approach, not a reskin of existing tools.

**In-process is the .NET-native path.** One NuGet. One line in Program.cs. Shares DI, auth, tenancy, and permissions with the host. No second deployment. This is how .NET developers prefer their integrations — it's why Hangfire won over Quartz, why SignalR won over socket libraries, why MediatR won over home-rolled buses.

**The paired Claude Skill (v1.0) is the "wait, what?!" moment.** When the NuGet package is installed, it can also emit a `.claude/skills/{appname}/` directory that teaches Claude how to use the MCP server well: ABP conventions, tenancy rules, permission handling, common patterns. Distributed together. Agent quality goes from "mechanical" to "fluent."

## Constraints

- Open source passion project — one developer's evenings and weekends.
- Sponsorship is a bonus, not a requirement. Success = real users installing and starring.
- .NET 9 / 10. Target ABP Framework v8+.
- Must respect ABP's existing permission system with zero configuration.
- Must not require ABP source modifications — works against stock ABP.

## Premises (agreed)

1. **Generate from ABP's own API description metadata, not OpenAPI.** Read `IApiDescriptionModelProvider` in-process (the same provider that powers `/api/abp/api-definition` and ABP's TS/C# proxy generators). This is richer than OpenAPI (typed DTOs, permissions, XML docs, localization) and idiomatic — we use ABP's own pipeline, not a parallel reinvention.
2. **In-process NuGet, not a sidecar server.** `builder.Services.AddAbpMcp()`. Shares everything with the host. This is the .NET-native path.
3. **Demo on a real ABP app first, generalize second.** Build against `eShopOnAbp` or a real app. Ship a video of an agent actually placing an order. Extract the reusable library afterward.
4. **Permission-aware tool exposure is non-negotiable.** Tool visibility respects the caller's granted permissions. Without this, zero enterprise adoption — and enterprise ABP users are most of ABP's user base.
5. **Tool exposure is opt-in via `[McpTool]` attribute, suppressed with `[McpIgnore]`.** Enterprise-safe default. A 30-entity app does not ship with 200 tools by accident; developers explicitly tag the services they want agent-accessible. Prevents the CRUD-noise explosion that breaks agent quality above ~50 tools. Naming mirrors ABP's own `[RemoteService(IsEnabled = ...)]` ergonomic.

## Landscape

- [ABP.IO published an MCP tutorial](https://abp.io/community/articles/building-your-first-mcp-server-with-.net-a-developers-guide-to-model-context-protocol-mfgqj2o4) — team is MCP-aware. Green light for a blessed integration.
- [`cyrilnoah1/abp-io-mcp`](https://lobehub.com/mcp/cyrilnoah1-abp-io-mcp) exposes ABP *framework management* (create modules, entities, tenants). This is **not competition** — it's for operating on ABP itself, not apps built on ABP. Worth collaborating with the author.
- Microsoft's `ModelContextProtocol.AspNetCore` SDK shipped March 2026 — foundation is ready.
- Generic OpenAPI→MCP tools exist in every language ([openapi-mcp-generator](https://github.com/harsha-iiiv/openapi-mcp-generator), [FastMCP](https://gofastmcp.com/integrations/openapi), [Stainless](https://xata.io/blog/built-xata-mcp-server)). They produce poor-quality servers because OpenAPI is too low-level.
- [Neon wrote an industry post](https://neon.com/blog/autogenerating-mcp-servers-openai-schemas) arguing against mechanical OpenAPI→MCP. Their conclusion: prune aggressively, rewrite descriptions. Our angle: don't start at OpenAPI in the first place.

## Approaches Considered

### Approach A: Reflection-based runtime (Weekend Hack)
- Startup reflection over `IApplicationService` implementations.
- Build MCP tool manifest dynamically at app start.
- Route calls through DI, pre-check permissions via `IAuthorizationService`.
- Effort: M (~2-3 weeks to demo-quality). Risk: Low.

### Approach B: Roslyn source generator (Real Architecture)
- Compile-time MCP bindings generated per app service. AOT-compatible.
- Attribute-driven overrides (`[McpTool]`, `[McpIgnore]`).
- Effort: XL (2-3 months). Risk: Medium.

### Approach C: A + LLM-enhanced descriptions + paired Claude Skill (Viral v1.0)
- Everything in A, plus a build-time step that uses an LLM to rewrite tool descriptions from XML docs, DTO semantics, and domain context.
- Output includes a `.claude/skills/{appname}/` directory distributed alongside the MCP endpoint.
- Effort: L (~4-6 weeks after A lands). Risk: Medium.

## Recommended Approach

**Ship Approach A lean. Decide on Approach C after the first 100 users.**

1. v0.1.0 = Approach A only: reflection-based runtime, in-process, permission-aware, ABP-idiomatic public surface, `/mcp/_discover` + `/mcp/_explain` diagnostics. Mechanical tool descriptions plus the `[McpTool(Description = "...")]` override for tools that warrant hand-tuning.
2. **Approach C is a post-launch experiment, not a v1.0 commitment.** LLM-enhanced descriptions and the paired Claude Skill are tempting, but premature without real agent-failure data. Once 100+ developers are using v0.1.0, we will have evidence: which tools agents fumble, which descriptions succeed, whether a paired skill actually changes outcomes. *Then* we decide whether to invest 4-6 weeks of build-time LLM tooling.
3. Keep Approach B (source generator) as a v2.0 consideration once runtime usage validates the product shape.

Why: nothing in A is wasted if Approach C ships later, and nothing in A is wasted if Approach C is the wrong investment. Optionality is cheap; speculative complexity is not. (This call updated 2026-04-25 based on community feedback that LLM-description optimization is premature without agent-failure data.)

## Public API surface (v0.1)

ABP-idiomatic ceremony, mirroring `AbpAspNetCoreMvcOptions.ConventionalControllers`:

```csharp
// Program.cs (raw ASP.NET Core path)
builder.Services.AddAbpMcp(opts =>
{
    opts.Path = "/mcp";
    opts.ExposedAssemblies.Create(typeof(MyAppApplicationModule).Assembly);
});

// or, inside an AbpModule
public override void ConfigureServices(ServiceConfigurationContext context)
{
    Configure<AbpMcpOptions>(opts =>
    {
        opts.ExposedAssemblies.Create(typeof(MyAppApplicationModule).Assembly);
        opts.ToolNameNormalizer = ctx => /* custom */;
    });
}
```

Tag services for exposure:

```csharp
[McpTool]                                          // mirrors [RemoteService(IsEnabled=true)]
public class ProductAppService : ApplicationService, IProductAppService
{
    [Authorize("Products.Create")]
    public Task<ProductDto> CreateAsync(CreateProductDto input) => /* ... */;

    [McpIgnore]                                    // method-level opt-out
    public Task<int> InternalDiagnosticAsync() => /* ... */;
}
```

## Diagnostic endpoints (v0.1)

| Endpoint | Purpose |
|----------|---------|
| `GET {path}/_discover` | Lists every exposed tool with name, description, service, method, parameters, required permissions, JSON schema. |
| `GET {path}/_explain?service=Product` | For each method in matching services: `included: true/false` + reason text (which rule decided). Cures "why isn't my tool showing up?" |

Both inherit the host's auth — anonymous access is allowed only when `AllowAnonymous = true`.

## Exposure rules (v0.1, layers 1, 4, 5 of the v1.0 model)

Strict precedence, highest first:
1. **Attributes:** `[McpIgnore]` (always wins) > `[McpTool]` on method > `[McpTool]` on class
2. *(reserved for v0.2: type/method predicates)*
3. *(reserved for v0.2: include/exclude glob + regex)*
4. **Conventions:** must be a public instance method on a type discovered via ABP's api-definition (i.e. an `IApplicationService` exposed as a controller).
5. **Default exposure policy:** opt-in. No attribute → not exposed.

Per-tool kill switch via `AbpMcpOptions.DisabledTools` (hot-reloadable through `IOptionsMonitor`). v0.2 backs this with ABP Settings Management for cross-instance coordination.

## Open Questions

1. **Streaming and long-running tools** — how does `AbpMcp` handle app service methods that take > 30s? Background jobs? SignalR streaming? MCP's streamable HTTP transport matters here.
2. **Authentication model** — for remote MCP (not stdio), how does the agent authenticate? Bearer token from ABP's existing token endpoint? OAuth flow? This is the single most likely blocker for enterprise adoption.
3. **CRUD service noise** — ABP's `ICrudAppService<T>` produces 6-7 tools per entity. A medium app = 100+ tools. Do we expose them all? Offer a filter attribute? Group under a "bulk" meta-tool? Agents get worse with more tools.
4. **DTO validation errors** — how do validation failures surface to the agent? As structured errors or free-text? Agents recover better from structured errors.
5. **Multi-tenancy UX** — if the agent is running for a specific tenant, how is the tenant context set? Header? Authenticated claim? Explicit tool argument? This affects the shape of every tool.
